Introduction

This support module provides ASYN drivers to control and acquire data from the ALS high speed digitizer.  The module consists of C source, a database, a python client, a command line application and engineering screens. 

Database Development

The example application distributed with this support module comes with a database containing all the following records and EDM screens to display them.  All record names begin with the macros $(P)$(R).  Only the portion of the record name following these macros is shown in the following tables.

In "long segment" acquisition mode the waveform consists of 256 segments of 498 samples each so the NORD field will be 127488.  In "short segment" acquisition mode the waveform consists of 2048 segments of 50 samples each so the NORD field will be 102400.

The mbbiDirect record ADC:busy, with DTYP=asynUInt32Digital,  INP="@asyn($(PORT) 0x0003)", polls the waveform recorders and reads back the waveforms from all that have completed acquisition.  Typically SCAN=".5 second" or perhaps ".2 second".  Bit 7 corresponds to input 1, bit 0 to input 8,

Segmented acquisition applications usually fall into one of the following two cases.
  1. Repetitive measurements in a circular machine
    In this case the segment intervals are set to an integer multiple (1 to N) of the number of ADC clocks per revolution.  Either short or long segment acquisition is used depending on the length of the signal of interest.
  2. Lower effective sampling frequency to obtain data over a longer interval
    In this case the segment intervals are set to provide a lower effective sampling frequency and usually short segment acquisition is used.  Often all the samples in a segment are then averaged to provide a single value at each effective sampling point.

FPGA update on reconnect

The IOC attempts to automatically restore FPGA settings following an FPGA restart.  This is achieved through a calcout record ($(P)$(R)reconnChk_) that monitors the STAT field of a periodically-scanned readback record ($(P)$(R)sysmonTrig_) and causes a number of event and fanout records to process when that status returns to 0 or when the FPGA 'power up' condition is detected.  The fanout records in turn cause the various input and output records to process and read from or send their values to the FPGA.

Using high speed digitizer support in an application

Several files need minor modifications to use high speed digitizer support in an application.

  1. Add the full path to the high speed digitizer support directory to the application configure/RELEASE file:
    HSD=xxxx/modules/instrument/highSpeedDigitizer/<release>
    Where <release> is the release number of of the high speed digitizer support. You'll need to provide a similar line for the ASYN support module.
  2. Add instrument support to application database definition file
    The application database definition file must include the database definition files for the high speed digitizer. There are two ways that this can be done:
    • If you are building your application database definition file the old-fashioned way from an xxxInclude.dbd file you include the additional database definitions in that file:
      include "base.dbd"
      include "highSpeedDigitizerSup.dbd"
    • If you are building your application database definition file from the application Makefile you specify the additional database definitions there:
      xxx_DBD += base.dbd
      xxx_DBD += highSpeedDigitizerSup.dbd
  3. Add the highSpeedDigitizerSup support libraries to the application
    You must link the highSpeedDigitizerSup support library and the ASYN support library with the application. Add the following lines to the application Makefile:
    xxx_LIBS += highSpeedDigitizer
    xxx_LIBS += asyn
    before the
    xxx_LIBS += $(EPICS_BASE_IOC_LIBS)
  4. Add a highSpeedDigitizerConfigure command to the application startup script.  The arguments must appear in the order shown below.
    1. The ASYN port name.
    2. The IPv4 address of the high speed digitizer FPGA.
    3. The priority at which the port threads will run.  A value of 0 results in the default medium priority.
    4. The number of event receiver clocks per second.
  5. Load a database file or files describing the records associated with the high speed digitizer.

High Speed Digitizer Python Class

This class uses the PyEpics support module and provides support for controlling a high speed digitizer.

Instantiation

class HighSpeedDigitizer(recordNamePrefix, inputList [, callback=None [, connection_timeout=2.0]])

  • recordNamePrefix(string) – the prefix of the high speed digitizer record names.
  • inputList (tuple or list) – digitizer channels to be controlled and read.
  • callback (callable, tuple, list or None) – user-defined callback function on changes to waveform records' value or state.
  • connection_timeout (float or None) – time (in seconds) to wait for connection before giving up.

Refer to the PyEpics support module documentation for details on the callback parameter.  A message is sent to sys.stderr and a ValueError exception is raised if a process variable can not be connected.

Methods

Most methods provide simple 1:1 mapping to corresponding channel access get or put operations.

The following 'get' methods take a single integer input argument specifying the waveform recorder from which the value is to be read. Don't attempt to get the bonded status of inputs that are the base of a bonding group (inputs 4 and 8).

  • getGain (input(integer))
  • getCoupling (input(integer)) – returns mbbi enumeration string (DC, AC, Calibration, Training)
  • getOffset (input(integer))
  • getPretriggerCount (input(integer))
  • getTriggerEdge (input(integer)) – returns mbbi enumeration string (Rising, Falling)
  • getTriggerMask (input(integer))
  • getTriggerLevel (input(integer)) – returns self-trigger level, in volts.
  • getBonded (input(integer))
  • getSegmode (input(integer))
  • getRearm (input(integer))
  • busy (input(integer)) - returns 1 if the  waveform recorder has been armed and is waiting for a trigger or is acquiring data, otherwise returns 0.

The input argument to each of the following 'put' methods can be one of three forms.  A single numeric argument puts the value to the specified waveform recorder. A numeric list or tuple argument puts the value to all specified waveform recorders.  An argument of None puts the value to all the waveform recorders specified when the high speed digitizer object was instantiated.  Don't attempt to put the bonded status of inputs that are the base of a bonding group (inputs 4 and 8).

  • putGain (input(integer, tuple, list or None), value(integer))
  • putCoupling (input(integer, tuple, list or None), value(integer or enumeration string))
  • putOffset (input(integer, tuple, list or None), value(float))
  • putPretriggerCount (input(integer, tuple, list or None), value(integer))
  • putTriggerEdge (input(integer, tuple, list or None), value(integer or enumeration string))
  • putTriggerMask (input(integer, tuple, list or None), value(integer))
  • putTriggerLevel (input(integer, tuple, list or None), value(float))
  • putBonded (input(integer, tuple, list or None), value(integer))  Attempts to bond the base input of a group (input 4 or 8) will be ignored.
  • putSegmode (input(integer, tuple, list or None), value(integer))
  • putRearm(input(integer, tuple, list or None), value(integer))
  • arm (input(integer, tuple, list or None))  Attempts to arm bonded channels will be ignored.
  • disarm (input(integer, tuple, list or None))

Access to waveform data is provided by the following two methods or by specifying a callback function that will be called whenever new data arrive.

  • waveformPV (input((integer)) – returns the PV object for the specified input.
  • waveformsDict() – returns a dictionary whose keys are the input numbers and values are the PV object for that input.

The restore() method restores all waveform recorder settings, other than rearm and arm settings, to the values they had when the instance was instantiated.
